# SP5 â€” Model Planning Agent Design

**Date:** 2026-05-11
**Status:** Approved for implementation

---

## 1. Goal

Replace the deterministic `default_training_plan` fallback with an LLM-based Model Planning Agent that synthesizes evidence from the experience pool, static ML rules, and the model registry to produce a structured `TrainingPlan`.

The LLM contributes judgment. The deterministic executor contributes reproducibility. The HITL gate ensures human oversight. These are three distinct layers with no overlap.

---

## 2. Architecture

### 2.1 Graph topology

```
START â†’ supervisor â†’ data_validator â†’ planner â†’ executor â†’ evaluator â†’ deployer â†’ END
```

| Node | Role | Type |
|---|---|---|
| `data_validator` | validates + profiles the dataset | LLM (ReAct) |
| `planner` | synthesizes evidence â†’ produces `TrainingPlan` | LLM (structured output) |
| `executor` | runs Optuna, writes to pool, writes to MLflow | Deterministic |
| `evaluator` | evaluates champion against baseline | LLM (ReAct) |
| `deployer` | registers model + HITL gate | LLM (ReAct) + interrupt() |

The existing `trainer` node is renamed to `executor`. No other topology changes.

### 2.2 Design principle

```
DatasetProfile + TaskMetadata
        â†“
Deterministic retrieval:
  - similar past experiences   (ExperiencePool.find_similar)
  - matched ML rules           (match_rules)
  - model registry             (get_models_for)
        â†“
PlannerContext  (assembled deterministically)
        â†“
LLM Model Planning Agent
  (synthesizes evidence â†’ auditable planning analysis + TrainingPlan)
        â†“
4-stage validation:
  1. Pydantic parsing of PlannerOutput
  2. _check_plan_integrity(output.plan)
  3. _check_evidence_references(output.evidence_used, ctx)
  4. _check_plan_exhaustiveness(output.plan, ctx.available_models)
        â†“
Deterministic Training Executor
        â†“
MLflow + ExperiencePool
```

The retrieval layer **narrows the planning context** by selecting relevant prior experiences. It does not prescribe which models to train. The LLM planner receives these experiences, together with static ML rules and the model registry, and synthesizes them into a structured `TrainingPlan`.

## 3. Contracts

### 3.1 `contracts/planner.py` (new file)

```python
from typing import Literal
from pydantic import BaseModel, Field
from mlops_agents.contracts.training import TrainingPlan


class EvidenceReference(BaseModel):
    source: Literal[
        "dataset_profile",
        "task_metadata",
        "experience",
        "rule",
        "registry",
    ]
    source_id: str | None = None
    # source_id must be None for dataset_profile and task_metadata sources.
    # For experience: must match a task_id in PlannerContext.similar_experiences.
    # For rule: must match a rule_id in PlannerContext.matched_rules.
    # For registry: must match a model_key in PlannerContext.available_models.
    summary: str


PlannerStatus = Literal["ok", "retry_ok", "failed"]


class PlannerOutput(BaseModel):
    planning_analysis: str = Field(
        description=(
            "Detailed public analysis generated by the LLM. "
            "This is an explanation artifact â€” not guaranteed access to "
            "the model's internal reasoning process."
        )
    )
    evidence_used: list[EvidenceReference] = Field(default_factory=list)
    risks_or_warnings: list[str] = Field(default_factory=list)
    plan: TrainingPlan
```

### 3.2 `contracts/training.py` â€” no change

`_check_plan_integrity` is a Pydantic model validator that fires for **all** `TrainingPlan` instantiations, including `default_training_plan`. Adding an exhaustiveness check there would break the deterministic planner (which may legitimately exclude ineligible models without populating `models_not_recommended`).

The exhaustiveness check is instead a standalone function in `agents/planner.py`:

```python
def _check_plan_exhaustiveness(plan: TrainingPlan, available_models: list[str]) -> None:
    accounted = (
        {c.model_key for c in plan.candidates}
        | {r.model_key for r in plan.models_not_recommended}
    )
    missing = set(available_models) - accounted
    if missing:
        raise PlannerError(
            f"These models are neither in candidates nor models_not_recommended: "
            f"{sorted(missing)}. Every available model must be explicitly included or rejected."
        )
```

Called only from `planner_node`, after `_check_plan_integrity` and `_check_evidence_references`.

### 3.3 `PlannerContext` â€” context assembled by deterministic code

```python
class CandidateResultCompact(BaseModel):
    model_key: str
    rank: int                    # 1 = best
    metric_value: float | None

class ExperienceSummary(BaseModel):
    experience_id: str           # task_id from ExperienceRecord
    similarity_score: float
    dataset_summary: str         # from experience_summary field
    models_trained: list[str]    # all model_keys tried
    best_model: str              # selected_solution.model_key
    validation_score: float
    candidate_results: list[CandidateResultCompact] = Field(default_factory=list)
    # compact ranking of all candidates; gives the LLM signal about runners-up and failures,
    # not just the winner. Populated from CandidateResult rows ordered by best_score desc.
    notes: str                   # short rephrasing for LLM consumption

class PlannerContext(BaseModel):
    current_dataset_profile: dict
    task_metadata: dict
    available_models: list[str]          # only models for this problem_type
    similar_experiences: list[ExperienceSummary]   # top-k, may be empty
    matched_rules: list[dict]            # [{"rule_id": ..., "summary": ...}]
```

`PlannerContext` is serialised to JSON and sent as the `HumanMessage` to the LLM.

## 4. Context builder (`agents/planner.py`)

The context builder runs **before** the LLM call. It is purely deterministic.

```
build_planner_context(profile, task_metadata, problem_type, k=5):
  1. get_models_for(problem_type)             â†’ available_models
  2. pool.find_similar(profile, problem_type, k)  â†’ list[RetrievalView] â†’ list[ExperienceSummary]
  3. match_rules({**profile, "problem_type": problem_type})  â†’ list[MLRule] â†’ matched_rules dicts
  4. return PlannerContext(...)
```

Cold-start (empty pool) is handled gracefully: `similar_experiences = []`. The LLM falls back to rules and registry only.

## 5. System prompt (`prompts/planner.yaml`)

```yaml
system: |
  You are the Model Planning Agent for an MLOps pipeline.
  Your job is to synthesize evidence and produce an auditable planning
  analysis and a structured TrainingPlan. You do not train models.

  WHAT YOU RECEIVE
  ----------------
  - current_dataset_profile: bucketed characteristics of the dataset
  - task_metadata: target column, horizon, frequency (if forecasting)
  - available_models: the only model keys you may use
  - similar_experiences: top-k past runs on similar datasets
  - matched_rules: expert ML rules that apply to this profile

  HOW TO USE EVIDENCE
  --------------------
  1. Start from dataset_profile + task_metadata (ground truth about the dataset)
  2. Use similar_experiences as empirical evidence â€” what worked, what failed, under
     which validation setup
  3. Use matched_rules as expert guidance â€” when rules and experiences conflict,
     resolve via the current dataset profile as tiebreaker
  4. Use available_models as the hard constraint â€” never invent model keys

  HARD CONSTRAINTS
  ----------------
  - Every model_key in candidates and models_not_recommended must come from available_models
  - Every available model must appear in either candidates or models_not_recommended (no omissions)
  - Every source_id you cite in evidence_used must exist in the context you received
  - For dataset_profile and task_metadata sources, leave source_id as null
  - For forecasting_settings, only use validation strategies:
      single_split, rolling_window, expanding_window
  - For tabular classification/regression, do not invent custom validation protocols
    unless the TrainingPlan schema explicitly supports them
  - Do not invent concrete hyperparameter search spaces. You may only provide
    hyperparameter or budget hints if the TrainingPlan schema explicitly supports them
  - The real search spaces live in the model registry and Optuna; you supply intent,
    not search ranges

  OUTPUT FORMAT
  -------------
  Produce a PlannerOutput with:
    planning_analysis: detailed explanation of your evidence synthesis and decisions
    evidence_used: list of EvidenceReference (source, source_id, summary)
    risks_or_warnings: list of strings â€” flag anything uncertain or conflicting
    plan: a valid TrainingPlan with every available model accounted for
```

## 6. LLM call + validation chain

```python
# agents/planner.py â€” planner_node(state)

ctx = build_planner_context(profile, task_metadata, problem_type)
llm = get_llm().with_structured_output(PlannerOutput)
last_error = ""

for attempt in range(2):
    try:
        messages = [
            SystemMessage(content=system_prompt),
            HumanMessage(content=ctx.model_dump_json(indent=2)),
        ]
        if attempt == 1:
            messages.append(HumanMessage(
                content=f"Your previous plan was rejected: {last_error}. "
                        "Please produce a corrected PlannerOutput."
            ))

        output: PlannerOutput = llm.invoke(messages)
        # Stage 1: Pydantic parsing â€” implicit in with_structured_output
        # Stage 2: _check_plan_integrity â€” fires inside TrainingPlan.__init__
        # Stage 3: evidence references â€” no hallucinated citations
        _check_evidence_references(output.evidence_used, ctx)
        # Stage 4: exhaustiveness â€” every available model accounted for
        _check_plan_exhaustiveness(output.plan, ctx.available_models)
        break  # all stages passed

    except (ValidationError, PlannerError, ValueError) as exc:
        last_error = str(exc)
        if attempt == 1:
            raise PlannerError(f"Planner failed after retry: {last_error}") from exc

retry_used = (attempt == 1)
```

### 6.1 `_check_evidence_references`

```python
def _check_evidence_references(refs: list[EvidenceReference], ctx: PlannerContext) -> None:
    exp_ids = {e.experience_id for e in ctx.similar_experiences}
    rule_ids = {r["rule_id"] for r in ctx.matched_rules}
    model_keys = set(ctx.available_models)

    for ref in refs:
        if ref.source in ("dataset_profile", "task_metadata"):
            if ref.source_id is not None:
                raise PlannerError(
                    f"{ref.source} reference must have source_id=None, got {ref.source_id!r}"
                )
        elif ref.source == "experience":
            if ref.source_id not in exp_ids:
                raise PlannerError(f"experience source_id {ref.source_id!r} not in context")
        elif ref.source == "rule":
            if ref.source_id not in rule_ids:
                raise PlannerError(f"rule source_id {ref.source_id!r} not in context")
        elif ref.source == "registry":
            if ref.source_id not in model_keys:
                raise PlannerError(f"registry source_id {ref.source_id!r} not in available_models")
```

## 7. Fallback + supervisor error handling

- `planner_node` raises `PlannerError` after two failed attempts
- The supervisor (or a thin error wrapper on the `planner` node) catches `PlannerError` and writes:
  ```python
  state["planner_status"] = "failed"
  state["error_message"] = str(error)
  ```
  then routes to END

- On success: `planner_node` writes:
  ```python
  state["planner_analysis"] = output.planning_analysis
  state["planner_evidence_used"] = [e.model_dump() for e in output.evidence_used]
  state["planner_warnings"] = output.risks_or_warnings
  state["training_plan"] = output.plan.model_dump()   # executor reads via TrainingPlan.model_validate
  state["planner_status"] = "retry_ok" if retry_used else "ok"
  state["planner_retry_used"] = retry_used
  ```

## 8. AgentState additions

New fields in `state/agent_state.py`:

```python
planner_analysis: str           # LLM-generated planning explanation artifact
planner_evidence_used: list     # list of EvidenceReference dicts
planner_warnings: list          # list of warning strings
planner_status: str             # "ok" | "retry_ok" | "failed"  (PlannerStatus)
planner_retry_used: bool        # True if second attempt was needed
```

`planner_analysis` is surfaced via SSE â€” gives the UI a "why this plan?" moment before training begins. `planner_retry_used` is exposed in state (not only in `planner_output_json`) so UI and tests can read it without parsing JSON.

**State serialization convention:** `planner_node` writes `state["training_plan"] = output.plan.model_dump()`. `executor_node` reads it back as `plan = TrainingPlan.model_validate(state["training_plan"])`. This keeps `AgentState` JSON-serializable throughout.

## 12. What the LLM does and does not do

**Does:**
- Select which registry-eligible models enter `plan.candidates` with `reason` per candidate
- Explicitly reject every non-selected model in `models_not_recommended` with a human-readable reason
- Prioritize candidates based on empirical evidence from similar experiences and static rules
- Reconcile conflicts between rules and past experiences using the current dataset profile
- Choose `forecasting_settings` (validation strategy + exog strategy per column) for forecasting problems
- Produce `planning_analysis` as an auditable explanation artifact
- Cite evidence via `evidence_used` (verifiable against the context it received)
- Flag risks and conflicts in `risks_or_warnings`

**Does not:**
- Train models
- Run Optuna
- Calculate metrics
- Select the final champion after training
- Access future validation exogenous values
- Invent model keys outside the registry
- Override deterministic validation safeguards

## 13. Academic framing

> The system stores a structured planning analysis generated by the LLM. This analysis does not claim to expose the model's internal reasoning process, but provides an auditable explanation of the evidence, trade-offs, and decisions that led to the generated `TrainingPlan`.

> The retrieval layer narrows the planning context by selecting relevant prior experiences, but it does not directly prescribe the models to train. The LLM planner receives these experiences, together with static ML rules and the model registry, and synthesizes them into a structured `TrainingPlan`.

---

## 14. Out of scope (V1)

- Multi-turn planning dialogue (LLM asks clarifying questions before producing the plan)
- Planner-initiated retrieval (LLM decides what to retrieve â€” retrieval is always deterministic)
- Search-space override generation by the LLM (hyperparameter ranges come from the registry)
- Planner fine-tuning or prompt optimization (static prompt in V1)
- Parallel candidate evaluation driven by the planner (executor always runs sequentially)

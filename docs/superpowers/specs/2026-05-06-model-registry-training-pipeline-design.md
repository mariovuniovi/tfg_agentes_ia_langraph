# Model Registry + Deterministic Training Pipeline — Design Spec

**Sub-project:** 3 of the MLCopilot-style model_agent foundation
**Date:** 2026-05-06
**Branch:** feature/model-registry-training (TBD)

---

## Goal

Replace the LLM-driven trainer with a deterministic, multi-candidate training executor that takes a structured `TrainingPlan` and produces:

1. A trained champion model (artifact + MLflow run).
2. Train/test split files for the evaluator agent.
3. An *experience record* JSON capturing every candidate tested (including failures and rejected models) — the input format consumed by SP4's experience pool.

No LLM agent runs during training in SP3. The executor accepts a `TrainingPlan` from any source — hardcoded defaults, test fixtures, or (future SP5) the planner agent. SP3 is the **infrastructure layer** the agent will plug into.

---

## Architecture

```
TrainingPlan (input)  ─┐
                       ├──►  TrainingExecutor  ──►  ChampionResult + ExperienceRecord (output)
Dataset + Schema  ─────┘         │
                                 │ uses
                                 ▼
                        ┌────────────────────────┐
                        │ ModelRegistry (YAML)   │
                        │   ├─ factories.py      │
                        │   └─ search_spaces.py  │
                        └────────────────────────┘
                                 │
                                 │ trains via
                                 ▼
                        ┌────────────────────────────┐
                        │ Optuna (HP search)         │
                        │ MLflow (parent + children) │
                        │ statsforecast / skforecast │
                        │ sklearn / LGB / XGB / CB   │
                        └────────────────────────────┘
```

The executor never reads from raw inputs — it consumes the canonical processed CSV produced by the data_validator (`state["dataset_path"]`).

## Section 1 — Model Registry

### 1.1 YAML manifest: `src/mlops_agents/models/registry.yaml`

One entry per supported `model_key`, scoped by `problem_type`. The YAML carries both executable references (`factory`, `search_space.name`) and agent-readable metadata (`use_when`, `avoid_when`, `notes`, `complexity_rank`) so SP5 can render this content into the planner prompt without code changes.

**Structured `search_space` (declarative, validatable):**

```yaml
- model_key: lightgbm_regressor
  problem_type: regression
  family: gradient_boosting
  complexity_rank: 3                 # lower = simpler (used in champion tie-break)
  library: lightgbm
  factory: build_lightgbm_regressor

  search_space:
    name: lightgbm_regressor_space
    params:
      n_estimators: { type: int, low: 100, high: 1000 }
      learning_rate: { type: float, low: 0.005, high: 0.2, log: true }
      num_leaves: { type: int, low: 15, high: 127 }

  default_params:
    n_estimators: 500
    learning_rate: 0.03
    num_leaves: 31
    random_state: 42

  requires:
    min_rows: 500
    needs_scaling: false
    supports_missing: true
    supports_categorical: partial

  use_when:  [tabular_regression, nonlinear_relationships, medium_or_large_dataset]
  avoid_when: [very_small_dataset, strict_interpretability_required]
  notes: >
    Strong default for tabular regression with nonlinear patterns.
```

`search_space.params[name]` types: `int`, `float`, `categorical`. Float supports `log: true`. Int may carry `step`. Categorical carries `choices: [...]`.

This declarative form drives **two things**:
1. The Optuna `suggest_*` calls (auto-generated by `search_spaces.py`).
2. The `search_space_override` validator (parameter-by-parameter range/choice subset checks).

### 1.2 Python factories: `src/mlops_agents/models/factories.py`

```python
FACTORY_REGISTRY: dict[str, Callable[..., Any]] = {
    "build_logistic_regression":      build_logistic_regression,

    "build_auto_arima":               build_auto_arima,
    # ... all 20 factories

```

Factories return either a sklearn-compatible estimator (tabular and supervised forecasting) or a forecaster object (`statsforecast.StatsForecast` for statistical forecasters, `skforecast.ForecasterRecursiveMultiSeries` for supervised forecasters). Forecasting factories accept `task_metadata` so they can read `forecast_horizon`, `series_id_columns`, and `frequency`.

### 1.3 Search-space builder: `src/mlops_agents/models/search_spaces.py`

A single generic builder reads the structured YAML and emits Optuna suggestions:

```python
def build_suggest_fn(spec: SearchSpaceSpec) -> Callable[[optuna.Trial], dict]:
    def suggest(trial: optuna.Trial) -> dict:
        out = {}
        for name, p in spec.params.items():
            if p.type == "int":
                out[name] = trial.suggest_int(name, p.low, p.high, step=p.step or 1)
            elif p.type == "float":
                out[name] = trial.suggest_float(name, p.low, p.high, log=p.log or False)
            elif p.type == "categorical":
                out[name] = trial.suggest_categorical(name, p.choices)
        return out
    return suggest
```

No hand-written `suggest` functions — the YAML is the source of truth. If a future model needs conditional sampling, we add it as a special case here.

### 1.4 Loader: `src/mlops_agents/models/loader.py`

```python
class SearchParamSpec(BaseModel):
    type: Literal["int", "float", "categorical"]
    low: float | int | None = None
    high: float | int | None = None
    step: int | None = None
    log: bool = False
    choices: list[Any] | None = None

class SearchSpaceSpec(BaseModel):
    name: str
    params: dict[str, SearchParamSpec]

class ModelSpec(BaseModel):
    model_key: str
    problem_type: Literal["classification", "regression", "forecasting"]
    family: str
    complexity_rank: int
    library: str
    factory: str
    search_space: SearchSpaceSpec
    default_params: dict[str, Any]
    requires: dict[str, Any] = {}
    use_when: list[str] = []
    avoid_when: list[str] = []
    notes: str = ""

    @field_validator("factory")
    def factory_must_be_registered(cls, v):
        if v not in FACTORY_REGISTRY:
            raise ValueError(f"Unknown factory: {v!r}")
        return v

def load_registry(path: Path = REGISTRY_PATH) -> dict[str, ModelSpec]: ...
def get_models_for(problem_type: str) -> list[ModelSpec]: ...
def get_model(model_key: str) -> ModelSpec: ...   # raises KeyError if unknown
```

The loader rejects unknown factory names — security guarantee. No arbitrary code execution from YAML.

### 2.2 `contracts/training.py`

```python
class TrainingPlanCandidate(BaseModel):
    priority: int                         # unique within a plan
    model_key: str
    initial_hyperparameters: dict[str, Any] = Field(default_factory=dict)
    search_space_override: dict[str, Any] | None = None
    requested_trials: int | None = None   # advisory — see Section 3.3
    reason: str = ""

class RejectedModel(BaseModel):
    model_key: str
    reason: str

class TrialBudget(BaseModel):
    total_trials: int = 60
    allocation_strategy: Literal["priority_weighted", "equal"] = "priority_weighted"
    max_trials_per_candidate: int = 30
    min_trials_per_candidate: int = 5

class TrainingPlan(BaseModel):
    problem_type: Literal["classification", "regression", "forecasting"]
    metric_to_optimize: str | None = None        # default per problem_type if None
    candidates: list[TrainingPlanCandidate]
    models_not_recommended: list[RejectedModel] = []
    trial_budget: TrialBudget = Field(default_factory=TrialBudget)
    validation_strategy: dict[str, Any] | None = None     # SP5 may override defaults
    forecasting_settings: dict[str, Any] | None = None    # SP5 may set lags, multi_series_strategy, etc.
```

**Validation rules** (run in `model_validator`):
- Every `model_key` exists in the registry.
- Every `model_key` matches `problem_type`.
- Priorities are unique.
- `search_space_override` is a *strict subset* of the registry's space (rules in 3.2).

### 3.2 `search_space_override` validation rules

For each parameter `p` in the override:

1. **`p` must exist** in the registry's `search_space.params`. Unknown name → `ValueError`.
2. **Type compatibility**: override entries must match the YAML param type.
3. **Range/choice subset check**:
    - `int` / `float`: override is either a list of explicit values *all inside* `[low, high]`, or a `(low, high)` pair where both endpoints are inside the registry range. Disjoint or out-of-range → `ValueError`.
    - `categorical`: override choices must all be in the YAML `choices` list.
4. After validation, the executor builds a *narrowed* `SearchSpaceSpec` and feeds it to `build_suggest_fn`.

The executor logs the effective (narrowed) search space to the MLflow child run.

### 3.3 Trial budget allocation

`requested_trials` is **advisory**. Authoritative source: `trial_budget.total_trials`. Algorithm:

1. Compute base budget per candidate from `total_trials` + `allocation_strategy`:
    - `priority_weighted`: weights inverse to priority — priority 1 gets the largest share. Specifically, with N candidates, candidate of priority `k` gets `total_trials × (N + 1 - k) / sum(1..N)` (rounded), then redistribute remainder to priority 1.
    - `equal`: split evenly (remainder to priority 1).
2. If `candidate.requested_trials` is set, replace base with `requested_trials` *as a preference*.
3. **Clamp** every candidate to `[min_trials_per_candidate, max_trials_per_candidate]`.
4. **Renormalize**: if the sum of clamped values exceeds `total_trials`, scale down proportionally (keeping `min_trials_per_candidate` floor). If less, leave the gap — we never trade extra trials between candidates.

### 3.4 Step-by-step flow

1. **Build dataset profile** via `profiler.build_dataset_profile(dataset_path, task_metadata)`. Used in the experience record and (later) by SP5 for retrieval.

2. **Split** the canonical dataset using `splitter.py` (Section 4). Write three files to `output_dir`:
    - `<schema>_train_pool.csv`
    - `<schema>_test.csv`
    - `<schema>_split_metadata.json` — split kind, sizes, dropped series (if any), seeds.

3. **Allocate trials** (3.3).

4. **Open MLflow parent run** with name `pipeline_<timestamp>`.

5. **For each candidate (priority order):**
    a. Open MLflow child run named after `model_key`.
    b. Resolve effective search space: registry default ∩ `search_space_override` (validated above).
    c. Compose Optuna objective:
        - **classification**: StratifiedKFold (n_splits=5 default) on train_pool. Score = mean folds of macro F1.
        - **regression**: KFold(shuffle=True, n_splits=5) on train_pool. Score = mean folds of negative RMSE (Optuna maximizes).
        - **forecasting**: single temporal holdout inside train_pool — last `forecast_horizon` periods of each series form the **candidate validation slice**; everything before is candidate train. Score = chosen metric on validation slice.
    d. Run Optuna study with the candidate's allocated `n_trials`. Sampler = TPE; pruner = MedianPruner.
    e. **Per-trial failures**: caught inside the objective; the trial is marked FAIL by Optuna and the study continues with the next trial. We do **not** retry the same trial.
    f. **Candidate failure** = no successful trial produced a valid score. On candidate failure: retry **once** with `default_params` (no Optuna). If still failing → mark `status: "failed"`, log error info, close child run, continue to next candidate.
    g. On success: record `best_params`, `best_score`, `best_score_std`, `n_trials_used`, `duration_s`.
    h. Log to MLflow child run:
        - **Always**: `best_params`, `best validation metrics`, the full Optuna trials dataframe as a CSV artifact, the effective search space as a JSON artifact.
        - **Only if `settings.log_non_champion_models == True`**: a fitted candidate model (re-fit on train_pool with `best_params`).
        - The champion-elect model is logged after step 7.

6. **Pick champion candidate**:
    - Filter to `status == "successful"`.
    - Identify the best mean validation score (direction-aware: maximize for F1/accuracy, minimize for RMSE/MAE/MAPE/sMAPE).
    - **Tie tolerance** (`settings.tie_tolerance_relative`, default `0.01`):
        - For *maximize* metrics (F1, accuracy, R²): candidate is tied if `candidate_score >= best_score * (1 - tolerance)`.
        - For *minimize* metrics (RMSE, MAE, MAPE, sMAPE): candidate is tied if `candidate_score <= best_score * (1 + tolerance)`.
        - Among tied candidates, pick lowest `complexity_rank`. Further ties → pick lower `priority` integer.
    - If *all* candidates failed → `RuntimeError("All candidates failed: <list>")`.

7. **Retrain champion on full train_pool** with `best_params` and `random_state`. Save `models/champion.pkl`. Log to the *champion's child run*: tag `champion=true`, the champion model artifact, and (separately to the parent run) tag `champion_run_id=<child_run_id>`.

8. **Write the experience record** (Section 6) to `experience_pool/<task_id>.json`.

9. **Return** `TrainingResult`.

### 3.5 Default metrics per problem_type

| problem_type | metric_to_optimize (default) | direction | secondary reported |
|---|---|---|---|
| classification | macro_f1 | maximize | accuracy, roc_auc (binary only) |
| regression | rmse | minimize | mae, r2 |
| forecasting | rmse | minimize | mae, smape, mape (only when target has no zeros) |

`metric_to_optimize` from the `TrainingPlan` overrides this mapping if set.

### 3.6 Failure record format

In the experience record, candidate failures look like:

```json
{
  "model_key": "auto_arima",
  "status": "failed",
  "error_type": "ConvergenceError",
  "error_message": "ARIMA failed to converge after 50 iterations",
```

## Section 4 — Splitter

### File: `src/mlops_agents/training/splitter.py`

### 4.1 Tabular (classification, regression)

```
canonical.csv
   │
   ├── train_pool.csv (default 80%)
   └── test.csv       (default 20%)
```

- classification: `train_test_split(stratify=target, test_size=0.2)`.
- regression: `train_test_split(shuffle=True, test_size=0.2)`.

### 4.2 Forecasting

Two-level split:

```
canonical.csv
   │
   ├── train_pool.csv  (everything except final test horizon per series)
   └── test.csv        (final forecast_horizon periods per series)

Inside per-candidate Optuna (operates on train_pool only):
   │
   ├── candidate_train       (train_pool minus final validation horizon)
   └── candidate_validation  (final forecast_horizon within train_pool)
```

The disk artifacts are exactly `train_pool.csv` (which contains both candidate_train and candidate_validation) and `test.csv` (untouched until the evaluator).

### 4.3 Series length guards

Each series must satisfy:

```
series_length >= max(lags_used) + validation_horizon + test_horizon + min_train_points
```

Default `min_train_points = 30`. Policy:

- **Majority-too-short** (more than 50% of series fail the guard) → `ValueError` with explanation. The dataset is fundamentally too short for the requested horizon/lags.
- **Minority-too-short** → drop those series, log them in `split_metadata.json` under `dropped_series`, continue.

### 4.4 Returned metadata

`split_metadata.json` schema:

```json
{
  "split_kind": "stratified" | "random_shuffle" | "temporal_per_series",
  "n_train_pool": 1234,
  "n_test": 308,
  "test_size_ratio": 0.2,        // tabular only; null for forecasting
  "forecast_horizon": null,      // forecasting only; null for tabular
  "n_series_total": null,        // forecasting only
  "n_series_dropped": 0,         // forecasting only
  "dropped_series": [],          // forecasting only
  "random_state": 42
}
```

## Section 5 — Default plans (no agent in SP3)

### File: `src/mlops_agents/training/default_plans.py`

```python
def default_training_plan(
    problem_type: str,
    dataset_profile: DatasetProfile,
) -> TrainingPlan:
    """All eligible models from the registry with deterministic registry-order priority."""
    eligible = [m for m in get_models_for(problem_type) if _is_eligible(m, dataset_profile)]
    candidates = [
        TrainingPlanCandidate(
            priority=i + 1,                      # unique, registry order
            model_key=m.model_key,
            initial_hyperparameters=m.default_params,
            reason="default plan: registry-eligible",
        )
        for i, m in enumerate(eligible)
    ]
    if not candidates:
        raise ValueError(f"No eligible models for problem_type={problem_type}")
    return TrainingPlan(
        problem_type=problem_type,
        candidates=candidates,
        trial_budget=TrialBudget(allocation_strategy="equal", total_trials=60),
    )
```

`_is_eligible` enforces simple registry gates: `requires.min_rows`, `requires.needs_lag_features` against the dataset profile. SP5 will replace this with the planner agent.

## Section 7 — Graph integration

### `trainer_node` rewrite in `graphs/mlops_graph.py`

```python
def trainer_node(state: AgentState) -> Command:
    profile = build_dataset_profile(
        Path(state["dataset_path"]),       # canonical processed CSV from data_validator
        state["task_metadata"],
    )
    plan_dict = state.get("training_plan")
    if plan_dict:
        plan = TrainingPlan.model_validate(plan_dict)
    else:
        plan = default_training_plan(state["problem_type"], profile)

    result = run_training_plan(
        plan=plan,
        dataset_path=Path(state["dataset_path"]),
        target_column=state["task_metadata"]["target_column"],
        task_metadata=state["task_metadata"],
        output_dir=Path("data/processed"),
        mlflow_experiment=settings.mlflow_experiment_name,
    )

    return Command(
        goto="supervisor",
        update={
            "training_plan": plan.model_dump(),
            "train_pool_path": result.train_pool_path,
            "test_path": result.test_path,
            "split_metadata_path": result.split_metadata_path,
            "trained_model_path": result.champion_model_path,
            "training_run_id": result.mlflow_parent_run_id,
            "training_metrics": result.champion_metrics,
            "champion_candidate": result.champion_candidate,
            "experience_record_path": result.experience_record_path,
        },
    )
```

The trainer node reads `state["dataset_path"]` — by existing convention this is the canonical processed CSV written by `data_validator_node`. The trainer **never** reads `state["dataset_paths"]` (raw inputs).

## Section 11 — Acceptance criteria

SP3 is complete when:

1. The registry loads cleanly, all 20 model_keys present, all factories resolvable.
2. `search_space_override` validation rejects out-of-range, unknown-param, and disjoint overrides; accepts valid narrowings.
3. The executor runs an iris classification dataset end-to-end (default plan) and produces: train_pool, test, split_metadata, MLflow parent run with N child runs, champion model artifact, experience record JSON.
4. The executor runs a California Housing regression dataset end-to-end the same way.
5. The executor runs an AirPassengers forecasting dataset end-to-end the same way (single series).
6. The executor runs a multi-series forecasting dataset (e.g., M4 sample) end-to-end with `ForecasterRecursiveMultiSeries`.
7. A candidate that fails Optuna entirely (e.g., AutoARIMA on unstable data) does not crash the run; it appears in `models_tested` with `status: "failed"`.
8. The all-candidates-fail path raises `RuntimeError`.
9. Champion tie-break: when two candidates score within `tie_tolerance_relative`, the one with lower `complexity_rank` wins.
10. Test suite passes; no test calls a real LLM.
